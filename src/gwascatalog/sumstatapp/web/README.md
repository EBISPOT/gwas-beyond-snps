# GWAS Catalog Web Validator

Browser-based summary statistics validator using **PyScript** (Pyodide) and the
**EMBL Visual Framework**.

All validation runs **locally in the browser** — no data is uploaded to any server.

## Quick start

```bash
# 1. Build sumstatlib wheel and copy to web/dist/
python src/gwascatalog/sumstatapp/web/build.py

# 2. Serve locally and open in browser
python src/gwascatalog/sumstatapp/web/build.py --serve
# → http://localhost:8000
```

Or manually:

```bash
cd sumstatlib && uv build --wheel
mkdir -p src/gwascatalog/sumstatapp/web/dist
cp sumstatlib/dist/gwascatalog_sumstatlib-*.whl src/gwascatalog/sumstatapp/web/dist/
cd src/gwascatalog/sumstatapp/web
python -m http.server 8000
```

### Key decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Runtime | PyScript (Pyodide) | Reuses sumstatlib without changes; no server needed; Python team can maintain |
| Styling | EMBL Visual Framework CDN | On-brand; zero build step; pre-compiled CSS/JS |
| UI pattern | Multi-step wizard (HTML sections) | Matches existing TUI flow; "one thing per page" |
| Form controls | Native HTML5 `<input>`, `<select>`, `<details>` | No framework; accessible by default; minimal JS |
| File I/O | File System Access API + `<input type="file">` fallback | Read + write local files; graceful degradation |
| JavaScript | ~300 lines vanilla JS | Navigation, file picking, UI toggles only |
| Server | None (static files) | Deploy to any static host (GitHub Pages, S3, etc.) |

### Why not htmx?

htmx excels for server-rendered hypermedia applications. Since this app runs
entirely client-side (no server), htmx's `hx-get`/`hx-post` model doesn't
apply directly. However, the **htmx philosophy** is honoured:

- All content lives in semantic HTML (not generated by JS frameworks)
- Behaviour is declared in HTML attributes where possible
- Progressive enhancement over JavaScript-heavy SPAs
- No build step, no bundler, no node_modules

### Browser support

| Feature | Chrome/Edge | Firefox | Safari |
|---------|-------------|---------|--------|
| PyScript/Pyodide | ✅ | ✅ | ✅ |
| File System Access API (read+write) | ✅ | ❌ | ❌ |
| `<input type="file">` fallback (read) | ✅ | ✅ | ✅ |

## Deployment

The web app is a set of static files. To deploy:

1. Run `python build.py` to produce the wheel in `dist/`
2. Upload the entire `web/` directory to any static host:
   - **GitHub Pages**: push to a `gh-pages` branch
   - **AWS S3 + CloudFront**: sync the directory
   - **Any HTTP server**: `nginx`, `caddy`, `python -m http.server`

No server-side code is required. The only constraint is that the host must
serve files with correct MIME types (particularly `.whl` files).

## Development

```bash
# Rebuild after changing sumstatlib
python build.py --serve

# Skip rebuild (frontend changes only)
python build.py --skip-build --serve
```

## Limitations (MVP)

- **Initial load time**: Pyodide downloads ~15 MB on first load (cached after).
  A loading spinner is shown during this time.
- **Large files**: Files over ~100 MB may cause memory pressure in the browser.
  For very large datasets, the existing CLI tools remain the recommended approach.
- **Gzip support**: `.gz` files are decompressed in-browser using the
  `DecompressionStream` API (Chrome 80+, Firefox 113+, Safari 16.4+).
- **pydantic-core WASM**: Requires a recent Pyodide version that includes
  pydantic-core compiled for `wasm32`. If installation fails, check the
  Pyodide package index for compatibility.
